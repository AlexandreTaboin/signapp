📄 Application Signature Électronique

Documentation technique complète — fonctionnement, architecture et référence du code.

🎯 Objectif de l'application
Permettre à un gestionnaire (PC) de téléverser un PDF, d'y placer deux zones de signature (une par signataire), puis de l'envoyer à une tablette où les deux personnes signent à la main. Le serveur génère alors automatiquement un PDF final incrustant les signatures, téléchargeable par le gestionnaire.

Principes clés

  • Pas de base de données : chaque document est un fichier JSON storage/sessions/{doc_id}.json.
  • Deux rôles : pc (gestionnaire) et tablet (signataire), stockés en session PHP.
  • Protection CSRF sur toutes les mutations (POST).
  • Polling (toutes les 3 sec) côté PC et tablette pour synchroniser l'état.
  • Nettoyage automatique : cron quotidien + nettoyage opportuniste (1/20 requêtes).

🔄 Cycle de vie d'un document

Chaque document passe par plusieurs statuts stockés dans son JSON :

draft
Juste uploadé. Le PC place les zones de signature. Supprimé après 24h si abandonné.
↓
zones_placed
Les 2 zones sont définies. En attente d'envoi à la tablette.
↓
waiting_signatures
Visible sur la tablette, aucune signature encore.
↓
signed_by_1
Une personne a signé (pas forcément la P1). En attente du 2e.
↓
completed
Les 2 ont signé. Le PDF final est généré. Le PC peut le télécharger.
⚠️ Nettoyage automatique
  • draft > 24h → supprimé entièrement
  • completed téléchargé → cache effacé (PDF source, previews, PNG signatures) mais JSON + PDF final conservés
  • completed > 30 jours → tout supprimé, y compris PDF final

📦 Structure de données

Fichier JSON d'un document (storage/sessions/{doc_id}.json)

{
  "doc_id": "a3f2b8c1...",              // 32 chars hex, identifiant unique
  "created_at": 1730912345,
  "created_by": "gestionnaire",
  "status": "waiting_signatures",       // draft|zones_placed|waiting_signatures|signed_by_1|completed
  "original_filename": "contrat.pdf",

  // Identité des 2 signataires
  "p1_firstname": "Jean",
  "p1_lastname":  "Dupont",
  "p1_label":     "Jean Dupont",
  "p2_firstname": "Marie",
  "p2_lastname":  "Martin",
  "p2_label":     "Marie Martin",

  // Dimensions de chaque page (en pixels, issues du PNG)
  "pages": [
    { "num": 1, "width": 992, "height": 1403 },
    { "num": 2, "width": 992, "height": 1403 }
  ],

  // Zones de signature (coordonnées RELATIVES 0..1)
  "zones": [
    {
      "person": 1,           // 1 ou 2
      "page":   2,           // numéro de page
      "rel_x":  0.12,        // position x (0=gauche, 1=droite)
      "rel_y":  0.78,        // position y (0=haut, 1=bas)
      "rel_w":  0.20,        // largeur (fraction de la page)
      "rel_h":  0.07,
      "label":  "Jean Dupont"
    }
  ],

  // État des signatures
  "signed_1": true,
  "signed_2": false,
  "signature_1_file": "a3f2b8c1_p1.png",
  "signature_2_file": null,
  "signed_1_at": 1730913000,

  // Finalisation
  "sent_at":      1730912500,
  "completed_at": null,
  "final_pdf":    null,       // nom du fichier dans storage/final/
  "downloaded_at": null,
  "cache_cleaned": false
}
💡 Pourquoi des coordonnées relatives ?
Le PDF et son PNG de preview n'ont pas les mêmes dimensions. Utiliser des ratios (0..1) permet de calculer les positions sur n'importe quelle représentation (PNG pour l'affichage, millimètres pour le PDF final).

Fonctions utilitaires

generate_csrf_token() : string

Retourne le token CSRF de la session (en crée un si absent, bytes aléatoires hex de 64 caractères). Idempotent : même token pendant toute la session.

verify_csrf_token(?string $token) : bool

Compare le token reçu avec celui en session via hash_equals() (comparaison à temps constant, résistante au timing attack).

require_role(string $role) : void

Garde d'accès. Redirige vers /login.php si :

  • L'utilisateur n'est pas connecté
  • Son rôle ne correspond pas
  • Sa session a expiré (durée différente selon rôle)

json_response(array $data, int $code = 200) : void

Envoie une réponse JSON et termine le script. Sets Content-Type et http_response_code().

load_document(string $doc_id) : ?array

Charge un JSON de document avec verrou partagé (LOCK_SH). Retourne null si :

  • ID invalide (regex [a-f0-9], longueur < 16)
  • Fichier inexistant
  • JSON corrompu

save_document(string $doc_id, array $data) : bool

Sauvegarde avec verrou exclusif (LOCK_EX). Utilise ftruncate() + rewind() pour réécrire proprement.

get_pending_documents_for_tablet() : array

Parcourt tous les JSON et retourne ceux dont le statut est waiting_signatures ou signed_by_1. Triés par sent_at décroissant.

get_document_for_tablet(string $doc_id) : ?array

Charge un doc et retourne sa version "publique" pour la tablette : ajoute url absolue pour chaque page (vers /api/preview.php).

sanitize_name(string $s) : string

Nettoie un nom : trim, garde lettres Unicode/chiffres/espaces/- . ', max 50 caractères.

cleanup_document_cache(string $ doc_id) : void

Supprime les fichiers temporaires d'un document (PDF source, previews PNG, signatures PNG). Garde le JSON et le PDF final.

cleanup_storage() : array

Fonction principale de nettoyage. Retourne des stats :

[
  'cache_cleaned'    => N,   // nb de docs complétés dont cache a été vidé
  'expired_deleted'  => N,   // nb de docs > 30j supprimés
  'drafts_deleted'   => N,   // nb de brouillons abandonnés
  'orphans_deleted'  => N,   // nb de fichiers sans JSON
]
🎲 Nettoyage opportuniste
En bas de config.php : if (random_int(1, 20) === 1) cleanup_storage();
→ 1 requête sur 20 déclenche un nettoyage, en plus du cron quotidien.

🚪 Routeur & authentification

index.php — Routeur principal

Redirige selon le rôle en session :

if (empty( $ _SESSION['user'])) header('Location: /login.php');
elseif ( $ role === 'pc')     header('Location: /pc/index.php');
elseif ( $ role === 'tablet') header('Location: /tablet/index.php');

login.php — Connexion

Formulaire HTML + traitement POST :

  1. Vérifie le token CSRF
  2. Cherche le user dans USERS
  3. Vérifie avec password_verify()
  4. En cas de succès : session_regenerate_id(true) + création $ _SESSION['user'] + redirection
  5. En cas d'échec : délai de 0.5 s (anti-brute-force)

logout.php

Vide $ _SESSION, efface le cookie, détruit la session, redirige vers login.

📤 api/upload.php

POST auth: pc — Upload d'un PDF + création du document.

Paramètres (multipart/form-data)

ChampTypeRequis
csrf_tokenstring✓
pdffile✓ (PDF, ≤10 Mo)
p1_firstname, p1_lastnamestring✓
p2_firstname, p2_lastnamestring✓

Étapes

  1. Validations : méthode POST, CSRF, fichier reçu, taille, MIME, magic bytes %PDF-, noms sanitizés non vides.
  2. Génération ID : bin2hex(random_bytes(16)) → 32 chars hex.
  3. Déplacement fichier : move_uploaded_file() vers storage/pdf/{doc_id}.pdf.
  4. Conversion PDF→PNG :
    • Lecture du nombre de pages avec pdfinfo (regex Pages:\s+(\d+))
    • Conversion via Ghostscript en une seule commande (rendu 120 dpi, device png16m)
    • Vérification que chaque page a bien été générée
  5. Création du JSON avec statut draft.
  6. Réponse : {success, doc_id, pages: [{num, width, height, url}], p1_label, p2_label}
Pourquoi Ghostscript direct plutôt qu'Imagick ?
Plus rapide, moins de mémoire, pas besoin de policy.xml d'Imagick (qui bloque souvent les PDF par défaut pour des raisons de sécurité). Un seul appel système convertit toutes les pages d'un coup. 
// Commande Ghostscript utilisée
gs -q -dNOPAUSE -dBATCH -dSAFER -sDEVICE=png16m -r120 \
   -dTextAlphaBits=4 -dGraphicsAlphaBits=4 \
   -sOutputFile=storage/previews/{doc_id}/page_%d.png \
   storage/pdf/{doc_id}.pdf

🖼️ api/preview.php

GET auth: any — Sert une image de prévisualisation.

Paramètres

/api/preview.php?doc_id={32chars}&page={1-200}

Logique

  1. Vérifie la session (n'importe quel rôle connecté)
  2. Valide l'ID (regex hex) et le numéro de page (1-200)
  3. Lit storage/previews/{doc_id}/page_{N}.png
  4. Envoie avec Content-Type: image/png et cache privé 1h
Pourquoi ce proxy ?
Le dossier storage/ est bloqué par .htaccess → inaccessible en direct. Ce script vérifie l'authentification avant de servir l'image.

Body JSON

✍️ api/sign.php

POST auth: tablet — Reçoit une signature et, si les deux sont présentes, génère le PDF final.

Body JSON

{
  "csrf_token": "...",
  "doc_id": "...",
  "person": 1,                    // ou 2
  "signature": "data:image/png;base64,iVBORw0KG..."
}

Validations de la signature

  1. Regex du data URI : ^data:image/png;base64,(...)$
  2. Décodage base64 strict
  3. Taille binaire entre 100 octets et 2 Mo
  4. Magic bytes PNG : \x89PNG\r\n\x1a\n

Logique principale

// Sauvegarde PNG
$sig_file = SIGNATURES_PATH . '/' . $doc_id . '_p' .  $ person . '.png';
file_put_contents( $ sig_file, $bin);

// MAJ document
 $ doc['signed_'. $ person] = true;
 $ doc['signature_'. $ person.'_file'] = basename( $ sig_file);

// Transition statut
if ( $ doc['signed_1'] && $doc['signed_2']) {
     $ final = generate_final_pdf( $ doc);
    $doc['status']       = 'completed';
    $doc['completed_at'] = time();
     $ doc['final_pdf']    = basename( $ final);
}
elseif ($doc['signed_1']) $doc['status'] = 'signed_by_1';

Fonction generate_final_pdf(array $ doc) : ?string

Cœur de la génération PDF. Utilise TCPDF + FPDI :

  1. Décompression du PDF source via Ghostscript vers un fichier temp en PDF 1.4 : 
    gs -dCompatibilityLevel=1.4 -sDEVICE=pdfwrite \
   -sOutputFile=/tmp/fpdi_{id}.pdf {src.pdf}
    → FPDI ne supporte pas les PDF 1.5+ avec object streams. Cette étape rend le PDF importable.
  2. Import page par page avec setSourceFile + importPage + useTemplate
  3. Pour chaque zone : calcul des coordonnées absolues (rel_x * page_width), insertion du PNG de signature avec $ pdf->Image(), puis petit label "Signé : XXX" sous la zone
  4. Sauvegarde dans storage/final/{doc_id}_signed.pdf
  5. Nettoyage du fichier temp
⚠️ Piège connu
Si le PDF source contient des object streams (PDF 1.5+) ou est chiffré, FPDI lève une exception. D'où l'étape préalable avec Ghostscript qui convertit en PDF 1.4 simple. En cas d'échec de la décompression, on retombe sur le PDF original en fallback (mais ça peut planter plus loin).

📜 assets/js/pc.js

JS IIFE (function() { ... })() pour isoler les variables. Gère tout le workflow PC.

État central state

const state = {
    docId: null,          // doc en cours de préparation

Fonctions principales

upload-form submit (handler)

Envoie le FormData à /api/upload.php. En cas de succès, charge les pages, affiche l'étape 2, cache l'étape 1.

renderPages()

Crée dynamiquement les <img> de chaque page dans #pdf-pages, attache un click handler à chacune.

onPageClick(e, pageNum, container, img)

Calcule les coordonnées relatives du clic :

const rect = img.getBoundingClientRect();
const relX = (e.clientX - rect.left) / rect.width;
const relY = (e.clientY - rect.top) / rect.height;

Supprime l'ancienne zone de la personne active, ajoute la nouvelle, redessine.

redrawZones()

Nettoie toutes les .sig-zone du DOM et les recrée en positionnement absolu (pourcentages CSS).

checkZonesComplete()

Active le bouton "Envoyer à la tablette" uniquement si au moins une zone existe pour chaque personne (P1 et P2).

#btn-send-tablet click

Enchaîne 2 appels API :

  1. POST /api/set_zones.php (envoie le tableau de zones)
  2. POST /api/send_to_tablet.php (bascule en waiting_signatures)

Puis ajoute le doc à state.trackedDocs, réinitialise le formulaire, et lance le polling.

renderTracking()

Reconstruit la liste HTML des documents suivis dans #tracking-list. Pour chaque doc : titre, statut (waiting/pending/success), progression des 2 signataires, et bouton de téléchargement si complété.

startPolling() / pollAll()

Toutes les 3 secondes, pour chaque doc non complété, appelle GET /api/status.php?doc_id=... et met à jour l'état. Puis redessine la liste.

💡 Pattern
Le PC peut préparer un nouveau document pendant que des anciens attendent des signatures. Le polling continue tant qu'il reste des docs non complétés.

    const now = Date.now();
    if (now - lastTap < 300) {
        state.readingZoom = state.readingZoom === 1 ? 2 : 1;
        applyZoom();
    }
    lastTap = now;
});

#btn-start-signing click

Passe de l'écran de lecture à l'écran de signature : renderDocument() + updateSignerPanel().

renderDocument(doc)

Construit toutes les pages du doc avec les zones en surimpression (positionnées en pourcentages relatifs). Les zones ont un data-person pour identifier le signataire.

updateSignerPanel(doc)

Détermine qui doit signer :

  • Si P1 pas signé → P1 signe
  • Sinon si P2 pas signé → P2 signe
  • Sinon tout est fait, panneau caché

Met à jour la bannière et le titre avec le nom du signataire actif, et force le resize() du canvas via requestAnimationFrame (double RAF pour s'assurer que le DOM est peint avant le calcul de la taille).

#btn-validate click

Workflow complet de validation :

  1. Vérifie que le canvas n'est pas vide (sigInstance.isEmpty())
  2. Désactive le bouton, affiche "Envoi..."
  3. Positionne lockUntil à +5s pour bloquer le polling pendant la requête
  4. Convertit la signature en base64 via sigInstance.toBase64()
  5. POST /api/sign.php avec {csrf_token, doc_id, person, signature}
  6. En succès : toast "Signature enregistrée ✓" + retour à la liste (returnToList())
  7. En erreur : alert + réactivation du bouton

returnToList()

Réinitialise l'état, cache le panneau de signature, vide le conteneur, bascule sur l'écran liste et force un refresh immédiat.

startPoll()

Polling toutes les 3s. Saute si lockUntil est actif. Ne refresh que si on est en mode list (on ne perturbe pas l'utilisateur en train de signer).

Gestion du HiDPI et du redimensionnement

Le canvas a deux tailles distinctes :

  • Taille CSS (ce qu'on voit) : imposée par le layout
  • Taille buffer (résolution réelle) : multipliée par devicePixelRatio pour un rendu net
function resize() {
    // Sauvegarde du dessin courant
    const prev = hasContent ? canvas.toDataURL() : null;

    const rect = canvas.getBoundingClientRect();
    const dpr  = window.devicePixelRatio || 1;

    canvas.style.width  = rect.width + 'px';
    canvas.style.height = rect.height + 'px';
    canvas.width  = Math.round(rect.width  * dpr);
    canvas.height = Math.round(rect.height * dpr);

    // Scale pour dessiner en coordonnées CSS
    ctx.setTransform(dpr, 0, 0, dpr, 0, 0);

    // Fond ivoire + style de trait
    ctx.fillStyle = '#fffdf5';
    ctx.fillRect(0, 0, rect.width, rect.height);
    ctx.strokeStyle = '#111';
    ctx.lineWidth = 2.5;
    ctx.lineCap = 'round';
    ctx.lineJoin = 'round';

    // Restauration du dessin précédent
    if (prev) {
        const img = new Image();
        img.onload = () => ctx.drawImage(img, 0, 0, rect.width, rect.height);
        img.src = prev;
    }
}

Événements tactiles/souris

Les 3 handlers start/move/end sont branchés sur mouse* ET touch* avec { passive: false } pour pouvoir faire preventDefault() et empêcher le scroll pendant le dessin.

Export transparent toBase64()

Astuce pour avoir un PNG à fond transparent :

  1. Copie l'image dans un canvas temporaire
  2. Parcourt tous les pixels
  3. Si un pixel est "presque ivoire" (r>240 && g>235 && b>220), met son alpha à 0
  4. Exporte en data:image/png;base64,...
Limitation
Ce filtre peut aussi rendre transparents des traits très clairs. Pour une vraie signature au stylo noir, ça marche bien. Si tu veux des traits colorés, il faudrait travailler sur un canvas à fond déjà transparent dès le départ.
